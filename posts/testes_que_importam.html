Testes que Importam

Porque apenas cobertura de código não é suficiente para testes efetivos.

Apesar de já existir durante os anos 60/70, foi a partir dos anos 90 que houve a popularização do tema testes. Questões como automatização e Integração Contínua foram estabelecidas durante os anos 2000 com o amadurecimento do tema. Atualmente é muito difundido e comum, porém algumas questões ainda são debatidas.

Por que um teste pode não ser efetivo? ¶

Nem sempre o teste é efetivo, apesar de oferecer uma boa cobertura.

Isso pode gerar uma falsa segurança na suíte de testes, sendo que nem sempre cenários importantes são exercitados. Ou seja, não é apenas uma questão de cobrir linhas de código - e sim cobri-las com contexto. Dessa forma exercitar o código com fluxos bem definidos gera muitos benefícios.

Caminhos felizes ¶

O termo happy path (ou caminho feliz) indica o fluxo onde o código é executado sem erros, com saída esperada. É importante que haja um ou mais cenários de teste para o happy path, porém ele em si não é o suficiente. Para isto devemos cobrir caminhos alternativos.

Fora do caminho ¶

Caminhos alternativos representam erros ou falhas. É cobrir quando um usuário não envia a senha preenchida ou foge do padrão esperado. Estes fluxos são tão importantes quanto o happy path, pois são cenários funcionais da aplicação. Ou seja, devemos nos focar em diferentes casos de uso efetivos e compreender que o happy path é apenas mais um deles. Existem alguns cenários que podem ser considerados:

  • Validação de entrada
  • Casos limite (boundary testing)
  • Erros de rede/conexão
  • Dados corrompidos
  • Timeouts

Sobre limites ¶

Os testes de limite (boundary testing) tem como objetivo verificar como a aplicação se comporta com valores no limite dos dados válidos. O motivo é que são erros comuns, dessa forma os testes se tornam importantes. Considere:

  • Se valores entre 1 e 100 são válidos, logo verificar entradas ao redor dessa faixa devem ser feitos

Exemplo: Comprimento de senha ¶

Suponha que o sistema exija uma senha de 8 a 12 caracteres. Testes de limite seriam:

  • Senha de 8 caracteres (limite inferior, válido).
  • Senha de 12 caracteres (limite superior, válido).
  • Senha de 7 caracteres (inválido, abaixo do limite).
  • Senha de 13 caracteres (inválido, acima do limite).

Múltiplos cenários ¶

Os cenários são essenciais: a forma que os dados são fornecidos a aplicação é importante. Eles representam os usos reais da aplicação, onde o código é executado em situações práticas. Pois dessa forma conseguimos cobrir o que é essencial, criando cobertura a partir de casos de uso efetivos.

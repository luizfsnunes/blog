Você não precisa do else

Nem sempre ser explícito é ser legível

O controle do fluxo de um programa é uma das primeiras lições que temos ao aprender a programar. Existem muitas formas de realizar o controle e a mais comum é pelo uso do if. O if possui uma estrutura muito simples:

1if ok { 
2    // Bloco executado se verdade
3} else { 
4    // Bloco se falso
5}

Dessa forma é possível controlar um programa se baseando em expressões de forma bem simples. A regra é: desviar o fluxo se algo é verdadeiro, caso contrário desvia para o else. Porém, nem sempre o else é importante ou obrigatório.

Você lê mais código do que escreve ¶

Sim, isso é uma verdade. Ao se programar entender o que já existe é tão importante quanto criar. Escrever código limpo não faz diferença para o computador e sim para quem lê. Ter menos else encadeados com if é uma forma de simplificar o fluxo sem alterar comportamento, permitindo que outros programadores entendam facilmente os fluxos. A sequência das instruções não é clara, pois o leitor precisa ‘pular’ mentalmente entre blocos condicionais.

1ok := false
2if ok { 
3    // Este bloco não é executado	
4} else {
5    // Executa aqui
6}

Lemos a linha 1 e em seguida a linha 5, o que adiciona complexidade no entendimento. A complexidade aumenta em casos onde existem blocos if aninhados

1ok := false
2ok2 := true
3if ok { 
4    // Este bloco não é executado	
5} else {
6    if ok2 {
7        // bloco aninhado!	
8    }   
9}

Ou seja, temos múltiplos níveis de aninhamento. Isso é muito difícil de entender! Através da supressão do else podemos ter algo simples e que funcionalmente expressa a mesma ideia.

É importante entender que podemos ignorar o else na maioria dos cenários, apesar de existir cenários onde ele é mais útil.

Tudo é relacionado a quebra de fluxo ¶

A questão é que muitos dos cenários onde o else é utilizado não representam um desvio obrigatório de fluxo. Um exemplo bem claro é o seguinte:

1offset := 0
2if custom > 0 {
3    offset = custom 	
4} else {
5    offset = 10
6}

Neste caso não é uma questão de controle de fluxo, queremos apenas inicializar uma variável com valor padrão de acordo com uma configuração. É muito código para algo simples. Podemos simplificar da seguinte forma:

1offset := 10
2if custom > 0 {
3    offset = custom 	
4}

O comportamento não muda e é algo bem mais direto de entender. No caso do aninhamento dos blocos:

1ok := false
2ok2 := true
3if ok { 
4    // Este bloco não é executado	
5}
6if ok2 {
7   // bloco aninhado!	
8}

Muito mais simples! Apenas entendendo que o else não é obrigatório, melhorou a leitura do bloco.

Early Return é seu amigo ¶

Outro exemplo:

1for _, item := range collection {
2    if item.Draft {
3        // execute o bloco caso seja um draft	
4    } else {
5        // caso contrário, execute aqui
6    }
7}

Aqui podemos usar early return para evitar o bloco seguinte:

1for _, item := range collection {
2    if item.Draft {
3        // execute o bloco caso seja um draft 
4        continue
5    }
6    // execute aqui
7}

Em todos os casos o else pode ser ignorado, pois apenas o if deve ser executado. O early return é uma ferramenta muito útil para simplificações, pois geralmente o bloco else pode ser totalmente ignorado.

Então quando podemos usar? ¶

É claro que existem casos onde o else é essencial:

1if config.fileWriter { 
2    // caso tenha sido configurado
3} else {
4    // executa um writer padrão
5}

No exemplo acima a escrita sempre é executada, utilizando diferentes estratégias. A escrita em arquivo é uma exceção e o padrão é utilizado para outros casos. O else é necessário, pois faz parte do fluxo do programa como um todo. Nestes casos, o else é necessário.
